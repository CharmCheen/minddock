# 答辩演示要点（口语化风格）

---

## Q1：你的系统是怎么找到答案的？凭什么说这条结果是对的？

**背景**：评委质疑——RAG 系统说找到了相关段落，怎么证明不是瞎编的？

**我们的方案**：

不是把文档切成随机块，而是保留页-段-块层级关系。检索时能知道这条结果来自哪一页、哪个小节。

更重要的是，每条检索结果都带精确位置坐标：高亮句原文 + 字符级偏移（position_start / position_end）+ 证据参与状态（cited / revisited / copied）。

学生可以点开看原文，系统告诉他"高亮句是第 3 页第 2 段的第 5 句"，不是含糊地说"大概在这里"。

**实测数据**：
- 16 个文档全部重建成功，section_path 覆盖率 83%–98%

**演示建议**：输入"Attention is all you need"，Highlight 出现在第 3 页第 2 段，系统同时显示高亮原文 + 字符偏移 + 参与状态。

## Q2：你的评测指标是怎么来的？75% 的 hit@1 是不是过拟合了？

**背景**：评委质疑评测是否客观。

**我们踩过的坑**：初代 benchmark 有标注污染问题——出题的人和做题的人是同一拨人，标注不客观。

**我们的做法**：
1. 重新标注：清理了 6 处受污染数据，统一评测口径
2. 引入 Page-Mode 基线：纯分页检索，rerank 关掉，hit@1 = 0%

这样一来，75% 的 hit@1 是在基线为 0 的前提下取得的，改进是真实的。

**实测数据**：

| 指标 | 修订后 | Page-Mode 基线 |
|------|--------|--------------|
| hit@1 | **75%** | 0% |
| hit@3 | **85%** | 0% |
| hit@5 | **95%** | 0% |

**演示建议**：截图 show 修订前后的标注对比，说明"干净评测"的重要性。

## Q3：Compare（对比）功能怎么做？多主题查询怎么不打架？

**背景**：评委问——你说要"对比 A 和 B"，系统怎么知道两边各需要查什么？

**两个核心技术**：

1. **Query Decomposition**：系统先把"对比 A 和 B"拆成"查 A" + "查 B" 两个子查询，用正则匹配实现，避免两边混在一起查不出来

2. **Dual Retrieval**：语义检索 + 关键词检索并行跑，结果合并重排。单独一路可能漏，联合跑覆盖更全

**实测数据**：
- `compare_agentic_rag_vs_milvus`：修复前 Milvus 相关内容在 top-6 完全消失，修复后全部通过 ✅
- search/chat 主链无回归

**演示建议**：演示"对比 BERT 和 GPT 在摘要生成上的差异"，看系统能否分别召回两边的关键段落。

## Q4：怎么防止元数据干扰检索？

**背景**：评委问——如果用户想找"某位作者的文章"，但系统把 AUTHOR 区域的内容排在前面怎么办？

**Metadata-Aware Rerank（元数据感知重排）**：

给文档不同区域设了不同权重：
- AUTHOR / REFERENCE（作者、引用区）：**降权 -0.15**
- ABSTRACT（摘要区）：**降权 -0.05**
- 但如果用户明确说"我要找某位作者"，系统能识别并 bypass 降权

这个机制是加性的——只是调整分数，不改变排序公式，对正常内容查询不产生副作用。

**演示建议**：对比两个查询：
1. "attention mechanism survey" → ABSTRACT 被降权，不干扰
2. "Yoshua Bengio 的文章" → 显式 author intent，bypass 生效

## Q5：历史数据怎么处理？迁移会不会丢？

**背景**：评委问——系统上线后，以前存的文档要不要重新导入？

**安全 Re-ingest 迁移**：

不是直接改数据，而是：
1. 先对 Chroma 做带时间戳的完整备份
2. 给历史文档补上 section_path 和 semantic_type
3. 原子化删除+重建，保证迁移过程不出现不一致状态
4. 迁移后自动验证元数据完整性

**实测数据**：16/16 文档重建成功，迁移过程零数据丢失。

**演示建议**：截图 `reingest_metadata.py` 的执行日志，显示备份 → 迁移 → 验证全流程。

## Q6：你的分块策略到底好不好？怎么证明？

**背景**：评委问——凭什么说你的分块比固定 500 字分块好？

**我们的评测设计**：跑了 18 个真实查询 case，在相同 embedding 和检索配置下，对比 Page-Mode（固定分页）和 Structured+Hybrid+SoftV3（结构化分块）。

**实测数据**：

| 策略 | page_hit@1 | page_hit@5 | section_hit@1 | mrr_page | mrr_section |
|------|------------|------------|---------------|----------|-------------|
| Page-Mode 基线 | 0 | 0 | 0 | 0 | 0 |
| **Structured+Hybrid+SoftV3** | **0.667** | **0.944** | **0.611** | **0.759** | **0.690** |

Page-Mode 在 page/section/block_type 指标上全部为 0，说明没有结构化分块就没有这些指标的参考价值——这不是调参数调出来的，是分块策略本身就不同的结果。

**演示建议**：展示一个具体案例，同一篇 PDF 两种分块策略的检索结果差异。
